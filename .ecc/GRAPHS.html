code-review-graph: MCP Documentation for LLMs

For AI Agents (LLMs): This document explains how to interact with the code-review-graph MCP server. Use these tools before attempting naive greps or full-file reads to save context tokens and gain structural awareness.

1. What is code-review-graph?

It builds an AST (Abstract Syntax Tree) using Tree-sitter and stores a graph of nodes (functions, classes) and edges (calls, imports) in SQLite. When you request a code review or impact analysis, it computes the blast radius of the changes so you only read what is strictly necessary.

2. When to Use the Graph

  • Code Review: Use detect_changes + get_review_context to see what changed and only the surrounding context/callers.
  • Impact Analysis: Use get_impact_radius to understand what a file change affects down the line.
  • Code Exploration: Use semantic_search_nodes or query_graph to find code locations rather than using grep_search.
  • Architecture Understanding: Use get_architecture_overview or list_communities.

3. Key MCP Tools Reference

Tool Name Description Common Usage
detect_changes Analyzes uncommitted changes and returns risk scores & impact paths. Run this first when asked to "review the current changes".
get_review_context Gets the absolute minimum source snippets needed to review specific files. Run this after detect_changes to actually read the code.
get_impact_radius Finds all callers, tests, and dependents affected by a node or file. Run this when a user asks "what happens if I change X?"
query_graph Query relationships (e.g. callers_of, callees_of, imports_of, tests_for). Run this to trace dependencies without blind grepping.
semantic_search_nodes Full-text search (FTS5) for functions or classes. Run this to find where a specific logic is defined.
get_architecture_overview High-level view of the codebase structure and module clusters. Run this when starting work on an unfamiliar repo or planning architecture.
get_affected_flows Finds which execution paths (HTTP routes, CLI commands, etc.) are impacted. Run this before a deploy to understand risk.
refactor_tool Planning renames, finding dead code, analyzing coupling. Run this before major refactoring work.

4. CRITICAL Prerequisites & Gotchas

CRITICAL — Git Tracking Required: code-review-graph ONLY indexes files that are tracked by Git. If you create new files, you MUST run git add <files> BEFORE running code-review-graph build or code-review-graph update. Untracked files are completely invisible to the graph. This is by design (to exclude node_modules, bin, obj, etc.).
CRITICAL — Always Verify After Build: After running build or update, you MUST run code-review-graph status and confirm that the Nodes, Edges, and Files counts actually changed. If the numbers didn't change, something went wrong (usually: files not git-tracked). Never tell the user "it's working" without checking status output.
Database Location: The .code-review-graph/ directory MUST remain at the Git repository root. Moving it to a subdirectory will cause the tool to create a fresh empty database. Do not relocate it.
After Install Commands: code-review-graph install may auto-generate files like GEMINI.md, CLAUDE.md, etc. If you already have a central instructions file (e.g. AGENTS.md), delete the auto-generated duplicates immediately. Keep ONE source of truth.

5. Standard Workflow

# Step 1: Stage any new/modified files to Git
git add <new-or-modified-files>

# Step 2: Build or update the graph
code-review-graph build          # Full rebuild (use after major changes)
code-review-graph update         # Incremental (use for small changes)

# Step 3: VERIFY it worked
code-review-graph status         # Check Nodes/Edges/Files counts

# Step 4: Use MCP tools in your AI session
# The MCP server is auto-configured for Antigravity.
# Tools like detect_changes, semantic_search_nodes, etc. are available.
